
# Running the WRF Hurricane Matthew Tutorial on GMU's Hopper HPC

This guide outlines the steps to run a standard meteorological test case (Hurricane Matthew, October 2016) using the WRF Preprocessing System (WPS) and the WRF model on George Mason University's Hopper cluster.

It assumes you have already successfully compiled both `em_real` and WPS using the GNU/OpenMPI toolchain.

## Part 1: WPS (WRF Preprocessing System)

### 1. Request a Preprocessing Interactive Session

Do not run the preprocessing steps on the login nodes. Request an interactive compute node:

```bash
salloc --partition=interactive --time=02:00:00 --nodes=1 --ntasks=4

```

Load your required modules so the executables can run:

```bash
module purge
module load gnu9 openmpi4 netcdf-c netcdf-fortran
module load jasper libpng zlib

```

### 2. Download the Tutorial Data

Create a directory for your meteorological data and download the raw GRIB files for the Hurricane Matthew case.

```bash
mkdir -p ~/wrf_tutorial_data/matthew
cd ~/wrf_tutorial_data/matthew
wget https://www2.mmm.ucar.edu/wrf/TUTORIAL_DATA/matthew.tar.gz
tar -zxvf matthew.tar.gz

```

### 3. Configure and Run Geogrid

Navigate to your compiled `WPS` directory. Edit `namelist.wps` to match the tutorial dates and point to the static geographical data.

Open `namelist.wps` and update the `&share` and `&geogrid` sections:

```fortran
&share
 max_dom = 1,
 start_date = '2016-10-06_00:00:00',
 end_date   = '2016-10-08_00:00:00',
 interval_seconds = 21600,
/
&geogrid
 ! Ensure this path points to where the WPS_GEOG files are stored on Hopper
 geog_data_path = '/path/to/your/WPS_GEOG/'
/

```

Run the executable to interpolate the static terrestrial data onto your domain:

```bash
./geogrid.exe

```

*Look for: `Successful completion of geogrid.*`

### 4. Ungrib the Meteorological Data

Link the downloaded GFS GRIB files and the appropriate Variable Table (Vtable) to your WPS directory.

```bash
./link_grib.csh ~/wrf_tutorial_data/matthew/matthew/fnl
ln -sf ungrib/Variable_Tables/Vtable.GFS Vtable
./ungrib.exe

```

*Look for: `Successful completion of ungrib.*`

### 5. Metgrid (Interpolation)

Horizontally interpolate the ungribbed meteorological fields onto the domain generated by Geogrid.

```bash
./metgrid.exe

```

*Note: If you are using the GNU compiler, you may see warnings about `IEEE_OVERFLOW_FLAG IEEE_UNDERFLOW_FLAG IEEE_DENORMAL`. These floating-point exceptions are completely normal when interpolating masked boundary data and can be safely ignored as long as you see `Successful completion of metgrid.` at the end.*

## Part 2: Model Initialization (real.exe)

### 1. Link the Preprocessed Data

Navigate to your main WRF **run** directory. This directory contains all the necessary physics lookup tables required by the model. Symbolically link the `met_em` files generated by WPS here.

```bash
cd ~/WRF/run
ln -sf ~/WPS/met_em* .

```

### 2. Configure Namelist.input

Open `namelist.input` inside the `run/` directory and update the `&time_control` and `&domains` sections to perfectly match the dates and grid dimensions you set in `namelist.wps`.

```fortran
&time_control
 start_year                          = 2016,
 start_month                         = 10,
 start_day                           = 06,
 start_hour                          = 00,
 end_year                            = 2016,
 end_month                           = 10,
 end_day                             = 08,
 end_hour                            = 00,
 interval_seconds                    = 21600,
/

```

### 3. Initialize the Boundaries

Run `real.exe` to generate the initial and boundary condition files (`wrfinput_d01` and `wrfbdy_d01`). Because this is a lightweight process, it can be run in your current interactive session.

```bash
mpirun -np 1 ./real.exe

```

*Verification:* Check the bottom of the log file (`tail rsl.error.0000`). It must say `SUCCESS COMPLETE REAL_EM INIT`.

## Part 3: Running the Main Simulation via Slurm Batch

While interactive sessions are useful for preprocessing, the main model integration (`wrf.exe`) should be submitted to Hopper's dedicated compute queues.

### 1. Create the Slurm Batch Script

Ensure you are still in your `~/WRF/run` directory. Create a new file named `submit_wrf.slurm` and add the following configuration. This script requests 16 tasks and dynamically maximizes the memory request (4 GB per core) based on Hopper's standard compute node topology.

```bash
#!/bin/bash
#SBATCH --job-name=wrf_matthew
#SBATCH --partition=normal
#SBATCH --ntasks=16
#SBATCH --mem-per-cpu=4G
#SBATCH --time=0-06:00:00
#SBATCH --output=wrf_run_%j.out
#SBATCH --error=wrf_run_%j.err

# Clear the environment and load required modules
module purge
module load gnu9 openmpi4 netcdf-c netcdf-fortran

# Execute the model
mpirun ./wrf.exe

```

### 2. Submit the Job

Submit the script to the Slurm scheduler using the `sbatch` command:

```bash
sbatch submit_wrf.slurm

```

### 3. Monitor the Run

You can check the status of your job in the queue using:

```bash
squeue -u $USER

```

Once the job transitions to the `R` (Running) state, it will begin generating the standard `rsl.error.0000` log files in the `run/` directory. You can monitor the model's physics integration in real-time by running:

```bash
tail -f rsl.error.0000

```

*Verification:* The simulation is finished when the log reads `SUCCESS COMPLETE WRF`, and a `wrfout_d01_2016-10-06_00:00:00` NetCDF file is generated.

## Part 4: Output Verification with Python (Open OnDemand)

Once the simulation completes, the most efficient way to visually verify the output is by launching a JupyterLab session directly on Hopper's computing nodes. This avoids the need to download large NetCDF files to your local machine.

### 1. Launch JupyterLab via Open OnDemand

1. Navigate to the Hopper Open OnDemand portal at [https://ondemand.orc.gmu.edu](https://ondemand.orc.gmu.edu) (or the relevant Hopper OnDemand URL) and log in.
2. From the top navigation bar, click on **Interactive Apps** and select **JupyterLab**.
3. Request a standard compute node (1 core, 4GB RAM, 1-hour duration is sufficient for visualization) and click **Launch**.
4. Once the session is ready, click **Connect to Jupyter**.

### 2. Visualize the Data with `xarray`

In your JupyterLab interface, navigate to your `~/WRF/run` directory using the file browser on the left. Create a new Python 3 Notebook and run the following code to evaluate the 2-meter temperature field of your initial state:

```python
import xarray as xr
import matplotlib.pyplot as plt

# Load the WRF NetCDF output file
# Make sure the filename matches your actual generated output
file_path = 'wrfout_d01_2016-10-06_00:00:00'
ds = xr.open_dataset(file_path)

# Extract the 2-meter temperature (T2) at the first time step (Time=0)
# Convert from Kelvin to Celsius for easier interpretation
t2m = ds['T2'].isel(Time=0) - 273.15

# Generate the plot
plt.figure(figsize=(10, 6))
t2m.plot(cmap='magma', cbar_kwargs={'label': 'Temperature (°C)'})

plt.title('Initial 2m Temperature - Hurricane Matthew (Oct 6, 2016)')
plt.xlabel('West-East Grid Points')
plt.ylabel('South-North Grid Points')
plt.tight_layout()
plt.show()

```

If the code executes successfully and displays a geographically coherent temperature map (with cooler land masses and warmer ocean waters), your WRF compilation, preprocessing pipeline, and model integration are all fully functional.
